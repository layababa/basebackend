## 1. 审查发现报告（4 桶分类，含证据 file:line）

评审基准：xianyun 原始实现取自其提交 `aa10a87`（“接入 basebackend 公共 SDK”）的父提交 `aa10a87^`（删除前历史版本），逐点比对 SDK 抽取版本。

### 🔴 MUST FIX

**M1 — `main` 顶端 (`1fcad10`) SDK 无法编译，JitPack `main-SNAPSHOT` 构建必然失败**
- 证据：`config/AppConfig.kt:8` `import org.springframework.boot.autoconfigure.jackson.Jackson2ObjectMapperBuilderCustomizer`。
- 验证：`spring-boot-autoconfigure-4.0.3.jar` 中 **不存在任何 `jackson` 包类**（`unzip -l ... | grep jackson` 返回空）；Boot 4 已将 Jackson 自动配置整体移出该 jar。`./gradlew compileKotlin`（offline 与 online 均）失败：`Unresolved reference 'Jackson2ObjectMapperBuilderCustomizer'`、`modulesToInstall`/`failOnUnknownProperties`/`serializationInclusion` 全部 unresolved（AppConfig.kt:30-34）。
- 影响：boundary 规则 #5 要求接入方用 `main-SNAPSHOT`；当前 `main` 无法构建，任何接入方升级到最新快照都会拿不到产物。xianyun 之所以 pin 在旧 commit `3b03024`（build.gradle:30）正是被迫规避此破坏。
- 应改成（**第 3 轮调整：M1 删除动作并入 Jackson 全量迁移**）：删除 `kotlinModuleCustomizer()` bean 与失效的 `Jackson2ObjectMapperBuilderCustomizer`/`KotlinModule`/`registerKotlinModule` import。**与第 2 轮不同**：`@Bean objectMapper()` 也一并删除（不再手动提供 Jackson 2 mapper），改为依赖 Boot 4 自动装配的 Jackson 3 `JsonMapper`；SDK 内 18 个注入点同步迁移到 `tools.jackson.databind.json.JsonMapper`（详见 M2 全量迁移决策）。**M1 不再是“仅删 customizer”的孤立动作，而是 Jackson 全量迁移（Phase 4）的首步**——两个动作合并，避免“删 customizer 又保留 Jackson 2 objectMapper”这一过渡态（该过渡态在第 2 轮是为了最小侵入，第 3 轮用户选择直接全量迁移，故吸收）。

**M2 — `1fcad10` 的 Jackson 修复在 Boot 4 默认（Jackson 3）下无效，基于 Jackson 2 的过时假设**
- 证据：SDK `AppConfig.objectMapper()` 返回 `com.fasterxml.jackson.databind.ObjectMapper`（Jackson 2，AppConfig.kt:5,16-21）。xianyun 声明的是 **Jackson 2** kotlin module（`com.fasterxml.jackson.module:jackson-module-kotlin`，xianyun build.gradle:44）+ `spring-boot-starter-web` 4.0.3（build.gradle:4,32）。
- 可独立坐实的硬事实（不依赖本 plan 自身论断）：① `spring-boot-autoconfigure-4.0.3.jar` 内**无任何 jackson 类**（`unzip -l | grep jackson` 空）；② `spring-boot-jackson2` 模块未被引入（无 Jackson 2 自动配置在 classpath）；③ Jackson 3 `tools.jackson` 仅在 `compileOnly`（运行期不一定到位）。
- ✅ **因果已独立坐实（Orchestrator WebSearch，2026-06-19）：Boot 4 默认装配 Jackson 3，MVC `@RequestBody/@ResponseBody` 默认走 Jackson 3 `JsonMapper`（`tools.jackson.*`），Jackson 2 仅为迁移保留。** 因此基于 Jackson 2 的 `Jackson2ObjectMapperBuilderCustomizer` 即便能编译也不会作用于默认 MVC 转换器。更关键的机制变化：**Boot 4 中 `HttpMessageConverters` bean 已弃用、直接贡献 `HttpMessageConverter` bean 不再被支持**，自定义转换器须改用 `Client/ServerHttpMessageConvertersCustomizer`；迁移期可用 `spring.jackson.use-jackson2-defaults=true` 对齐 Jackson 2 默认行为。结论：`1fcad10` 的 Jackson 2 customizer 路线在 Boot 4 下既不可编译（M1）又方向错误（M2）；**第 3 轮范围调整后修复方向 = 全量迁移到 Boot 4 默认 Jackson 3（决策 A′），删除所有手动 Jackson 2 mapper bean，SDK 内部统一注入 `JsonMapper`**。来源：Spring Boot 4.0 Migration Guide、spring.io “Introducing Jackson 3 support in Spring”（2025-10-07）、Jackson 3 Migration Guide（`com.fasterxml.jackson`→`tools.jackson`，annotations 仍 `com.fasterxml.jackson.annotation`，mapper 由可变 `ObjectMapper` 改为不可变 `JsonMapper`，异常改为非受检）。
- 影响：commit message 声称“从根本上消灭 Kotlin data class 反序列化 500”，但 `Jackson2ObjectMapperBuilderCustomizer` 在 Boot 4 已不在自动配置 jar 内（硬事实①），该 customizer 当前连编译都过不了（M1），更谈不上对 MVC 生效。
- 应改成 — **决策 A′（第 3 轮范围调整：从“保留 Jackson 2”改为“全量迁移到 Jackson 3”，由用户 Phase 5 闸门决定）**：
  - **不再手动提供任何 Jackson 2 mapper bean**。`AppConfig` 删除 `objectMapper()` 与 `kotlinModuleCustomizer()` 两个 bean，改为依赖 **Boot 4 自动装配的 Jackson 3 `tools.jackson.databind.json.JsonMapper`**（Boot 4 starter-web 默认 mapper，且会自动注册 classpath 上的 Jackson 3 kotlin module，从根本上解决 Kotlin data class 反序列化 500）。
  - SDK 内 **18 个**构造注入 `com.fasterxml.jackson.databind.ObjectMapper` 的类，迁移为注入 **`tools.jackson.databind.json.JsonMapper`**（Boot 4 自动暴露此 bean；`JsonMapper extends tools.jackson.databind.ObjectMapper`，两者皆可作为注入类型，统一用 `JsonMapper` 与 Boot 默认 bean 类型一致，避免注入歧义）。
  - 若 SDK 仍需自定义 mapper 行为（FAIL_ON_UNKNOWN_PROPERTIES=false、NON_NULL inclusion），用 Boot 4 支持的 **`JsonMapperBuilderCustomizer`**（而非已弃用的 `Jackson2ObjectMapperBuilderCustomizer`）；但当前两项行为可通过 `application.properties` 的 `spring.jackson.*` 表达，**优先用配置而非 customizer bean**，保持 AppConfig 轻量。注意：自定义 `HttpMessageConverter` 在 Boot 4 须用 `Client/ServerHttpMessageConvertersCustomizer`（`HttpMessageConverters` bean 已弃用），但本迁移不新增自定义转换器，无需触碰这条路径。
  - **API 兼容性核实（Jackson 2→3）**：方法名 `writeValueAsBytes` / `writeValueAsString` / `readValue` 在 Jackson 3 `JsonMapper` 上**保留同名签名**（来源：Jackson 3 迁移指南），故 `TrtcUserSigGenerator.writeValueAsBytes(sigDoc)` 序列化字节**不变**（同一 `LinkedHashMap` → 同一 JSON），UserSig 不漂移——**但必须由 T3.1 字节级回归测试护栏锁定**（见高风险点）。`@JsonProperty`/`@JsonInclude`/`@JsonIgnore` 等注解仍属 `com.fasterxml.jackson.annotation`（Jackson 2/3 共享），**7 个仅用注解的 DTO/model 文件不改**（实测：`MeetingDtos`/`MeetingParticipantDtos`/`MessageDtos`/`Admin`/`Bot`/`DeviceSession`/`User`；**AppConfig 不在此列**——它虽 import annotation `JsonInclude`，但该 import 仅服务于将被删除的两个 bean，删 bean 后 AppConfig 为零 jackson import）。
  - 🔴 **接入方（xianyun）Jackson 3 kotlin module 是本期 Jackson 迁移的硬发版前置（与 M3 secret-key 前置同级），不再仅“登记缺口”**：Boot 4 的 Jackson 3 `JsonMapper` **只有在运行期 classpath 真实存在 `tools.jackson.module:jackson-module-kotlin` 时才会装配 Kotlin module**，且自动注册本身有官方已知缺陷（spring-ai #5424，closed not planned）。实测：SDK build.gradle **当前完全没有** Jackson 3 kotlin module 依赖；xianyun build.gradle:44 是 Jackson 2 坐标 `com.fasterxml.jackson.module:jackson-module-kotlin`（Boot 4 BOM 解析为 J2），Boot 4 的 Jackson 3 `JsonMapper` **不会**采用它。**后果**：决策 A′ 删掉手动 mapper 后，若运行期没有 Jackson 3 kotlin module，原 `1fcad10` 想消灭的 Kotlin data class 反序列化 500 会**原样复发**。因此 xianyun 接入 `tools.jackson.module:jackson-module-kotlin` 升级为**硬发版前置**（顺序：xianyun 接入 J3 kotlin module → SDK Jackson 迁移版本上线），已写入任务分解前置（T1.0）、本期不实施清单依赖项、Critical Files。
  - **SDK 是否自带 kotlin module 的取舍（明确结论）**：库工程常规做法是把第三方框架 module 设 `compileOnly`/`optional`、由接入方提供 runtime，避免与接入方版本冲突——但此处 Jackson 3 kotlin module **运行期缺失会直接回归 500**，纯靠接入方提供存在“忘记接入即静默回归”的真实风险。**权衡建议**：① SDK 至少在 **`testRuntimeOnly`/`testImplementation`** 引入 `tools.jackson.module:jackson-module-kotlin`（否则 SDK 自己的 Jackson 反序列化测试也会复现 500，见 T3.5）；② 是否把它升为 SDK `implementation`（从根上消除接入方漏配回归）属可选强化——优点是“开箱不回归”，缺点是 SDK 强约束 Jackson 3 kotlin module 版本、与接入方潜在冲突。**本期默认采用“硬前置 + SDK testRuntimeOnly”路线**（接入方提供 runtime + 发版前置把关），并在 plan 标注 `implementation` 作为后续可选强化（若接入方屡次漏配再升级）。
  - 迁移过渡期 Boot 4 提供 `spring.jackson.use-jackson2-defaults=true` 可临时对齐 Jackson 2 默认行为（仅过渡兜底，不替代 J3 kotlin module 接入）。
- ✅ **注入歧义复核（全量迁移后）：无歧义。** 迁移后 SDK 不再有 Jackson 2 `ObjectMapper` bean；18 个注入点统一注入 Boot 4 自动装配的 `JsonMapper`（单一来源）。Jackson 2 与 Jackson 3 是不同包的不同类型，编译期即区分，运行期无同类型竞争 bean。
- 🔴 **Jackson 全量迁移高风险点（实测，迁移时逐项护栏）**：
  1. **`GlobalExceptionHandler.kt:3-6,158-186`（最高风险，含 2 处线上静默回归）**：`extractReadableMessage` 对 `HttpMessageNotReadableException.cause` 做 `is InvalidFormatException` / `is MismatchedInputException` / `is UnrecognizedPropertyException` / `is JsonParseException` smart-cast 分发，把反序列化错误映射为友好 400。迁移后 MVC 抛的是 **`tools.jackson.*`** 异常，这 4 个 `is` 判断（仍指向 `com.fasterxml.jackson.*`）会**全部 false** → 友好 400 message 提取静默失效、退化为通用错误。迁移目标（逐类确定）：
     - `InvalidFormatException` / `MismatchedInputException` / `UnrecognizedPropertyException` → `tools.jackson.databind.exc.*`（前 3 个 Evaluator 已确认正确）。
     - 🔴 **`JsonParseException`（:3 import / :181 smart-cast）→ `tools.jackson.core.exc.StreamReadException`**（Jackson 3 **没有** `JsonParseException`，对应类为 `StreamReadException`；若 plan 别处出现 `JsonProcessingException` 一律对齐为 `tools.jackson.databind.DatabindException`）。否则**编译失败**（unresolved import）。
     - 🔴 **`MissingKotlinParameterException`（:172 用 `cause::class.simpleName == "MissingKotlinParameterException"` 判“缺少必填字段”）→ Jackson 3 已删除该类并并入 `MismatchedInputException`（FasterXML/jackson-module-kotlin #617）**，迁移后该字符串恒不命中 → 友好提示「缺少必填字段」静默退化为通用「类型不匹配」（线上静默回归）。**落地改法**：去掉 `simpleName` 字符串判断，改为在 `is MismatchedInputException` 分支内基于 path 命中的非空 Kotlin 参数判定缺值。等价判断（仅依赖 Jackson 3 公有 API，无需 kotlin module 新异常类型耦合）：
       ```kotlin
       is MismatchedInputException -> {
           val field = cause.path.joinToString(".") { it.fieldName ?: "[${it.index}]" }
           if (field.isNotEmpty()) {
               // Jackson 3 下「缺少必填字段」表现为：MismatchedInputException 无具体值/无目标实例，
               // 且 message 含 null/missing 语义。MissingKotlinParameterException 已并入本类型。
               val msg = cause.message.orEmpty()
               val isMissing = msg.contains("missing", ignoreCase = true) ||
                   msg.contains("null value for creator", ignoreCase = true) ||
                   msg.contains("due to missing", ignoreCase = true)
               if (isMissing) "缺少必填字段「$field」" else "字段「$field」的类型不匹配"
           } else "请求参数类型不匹配"
       }
       ```
       > 实现期须由 T4.2 测试坐实：构造一个缺失非空 Kotlin 参数的请求体，断言返回「缺少必填字段」而非「类型不匹配」（Jackson 3 实际 message 文案以实测为准，上面三种 message 关键字为候选，落地时按 SDK 实跑结果收敛）。这是判断的 DoD。
  2. **`TrtcUserSigGenerator.kt:24 writeValueAsBytes(sigDoc)`**：UserSig 签名输入字节，迁移后字节必须逐字节不变（否则 HMAC 变化 → 所有下游 TRTC UserSig 失效、入会静默失败）。方法签名 Jackson 3 保留，但默认序列化行为差异（如 property 顺序、空值处理）需由 **T3.1 字节级回归测试**锁定——本测试是 Jackson 迁移的核心安全网。
  3. **18 个 `ObjectMapper` 注入点类型替换**：机械改动但量大，需确保每处 import + 构造参数类型 + `new TrtcUserSigGenerator(objectMapper)` 等传参点一致改为 `JsonMapper`。
  4. **7 个仅注解文件不可误改**：`MeetingParticipantDtos`/`MeetingDtos`/`MessageDtos`/`Admin`/`Bot`/`DeviceSession`/`User` 只用 `com.fasterxml.jackson.annotation.*`，Jackson 3 共享，**保持不变**，误改为 `tools.jackson` 会编译失败。**注意 AppConfig 不属于此列**：它的 `com.fasterxml.jackson.annotation.JsonInclude` import 只被 `objectMapper()`(:20) 与 `kotlinModuleCustomizer()`(:34) 两个待删 bean 引用，删 bean 后须**一并删除该 annotation import**，迁移后 AppConfig 为零 jackson import（误把它留在“注解保留”清单会残留无引用 import）。
  5. **build.gradle 依赖范围**：`tools.jackson.core:jackson-databind:3.0.4` 当前为 `compileOnly`（line 49）——SDK 是库工程（接入方 starter-web 带 Jackson 3 runtime），`compileOnly` 对编译足够、运行期由接入方提供，**保持 compileOnly 即可**，不必改 `implementation`（避免与接入方 Jackson 版本冲突）。同时应删除已不再被引用的 Jackson 2 `compileOnly`（lines 29-31 `jackson-annotations:2.20`/`jackson-databind:2.20.1`/`jackson-module-kotlin:2.20.1`）中的 `databind`/`module-kotlin` 两条——但 **`jackson-annotations` 必须保留**（7 个注解文件仍依赖）；保留 Jackson 2 annotations 与 Jackson 3 databind 共存是官方支持组合。**另须新增 `testRuntimeOnly`（或 `testImplementation`）`tools.jackson.module:jackson-module-kotlin`**：SDK 自己的 Jackson 反序列化测试（T3.5/GlobalExceptionHandlerTest）运行期需要 Jackson 3 kotlin module 才能实例化 `JsonMapper` 反序列化 Kotlin data class，缺失则 SDK 测试自身复现 Kotlin data class 反序列化 500。（是否进一步把 J3 kotlin module 升为 SDK `implementation` 以从根消除接入方漏配回归，见 M2/A′ 取舍结论：本期默认硬前置 + testRuntimeOnly，`implementation` 作为后续可选强化。）

## 10. 第 4 轮 Jackson 迁移事实修正记录

> 第 4 轮独立 Evaluator 用 jar `javap` / 官方源实证复核 Jackson 2→3 迁移块，发现 5 处硬错误（3 处会导致编译失败或线上静默回归）。M4 整块、AL-1~6（JsonMapper 类名/继承/方法名保留/注解共享/前 3 异常类 `tools.jackson.databind.exc.*`/compileOnly 策略/UserSig 字节护栏）经确认无误，未改。逐条记录：

- **MF-A1（最严重，线上静默回归）— `MissingKotlinParameterException` 在 Jackson 3 已删除**
  - 实证：jackson-module-kotlin 3.0 移除 `MissingKotlinParameterException`、并入 `MismatchedInputException`（FasterXML/jackson-module-kotlin #617）。
  - 现状代码（已读 `basebackend` SDK `GlobalExceptionHandler.kt:172`）：`val isMissing = cause::class.simpleName == "MissingKotlinParameterException"`，Jackson 3 下恒 false → 「缺少必填字段」退化为「字段类型不匹配」。
  - 改法：删除 `simpleName` 字符串判断，在 `is MismatchedInputException` 分支内按 path（命中字段）+ exception message 语义（`missing` / `null value for creator` / `due to missing`）判定缺值；落地 message 关键字以 SDK 实跑结果收敛。写入 M2 高风险点#1（含代码块）、§3.2 契约、§4 GlobalExceptionHandler DoD、T4.2、新增 `GlobalExceptionHandlerTest`（缺非空 Kotlin 参数 payload 断言「缺少必填字段」）作为 DoD。

- **MF-A2（编译失败）— Jackson 3 无 `JsonParseException`**
  - 实证（jar）：Jackson 3 对应类为 `tools.jackson.core.exc.StreamReadException`；`JsonProcessingException` 亦已删，对应 `tools.jackson.databind.DatabindException`。
  - 现状代码（SDK `GlobalExceptionHandler.kt:3,181`）：`import com.fasterxml.jackson.core.JsonParseException` + `is JsonParseException`。
  - 改法：迁移目标由模糊“tools.jackson.core.*”改为确切 **`tools.jackson.core.exc.StreamReadException`**；plan 内 `JsonProcessingException`（如出现）对齐 `tools.jackson.databind.DatabindException`。前 3 个异常类（→`tools.jackson.databind.exc.*`）保持不动（Evaluator 已确认正确）。写入 M2 高风险点#1、§3.2、§4 DoD、T4.2、Critical Files #2。

- **MF-A3（范围扩张引入的真实回归）— Boot 4 自动注册 J3 Kotlin module 不可靠**
  - 实证：Boot 4 Jackson 3 `JsonMapper` 仅在运行期 classpath 真实存在 `tools.jackson.module:jackson-module-kotlin` 时才装配它；自动注册有官方已知缺陷（spring-ai #5424，closed not planned）。SDK build.gradle 当前**无** J3 kotlin module；xianyun build.gradle:44 为 J2 坐标（Boot 4 BOM 解析为 J2），Boot 4 J3 `JsonMapper` 不采用它。
